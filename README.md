# enterprise-network-topology-packet-tracer
Here are the 8, in the order we hit them:

OSPF network statement typos — Router0 and Router1 each advertised the other router's LAN subnet instead of their own (copy-paste error), so no adjacency-breaking issue at first glance, but zero LAN routes propagated correctly until fixed.
Router2's serial interfaces had swapped IPs — assigned based on assumed port order, not actual cabling. Interfaces showed up/up (cabling fine), but OSPF formed zero neighbors because the two ends of each link were on mismatched subnets. Diagnosed by fixing hostnames first (all three routers still said "Router," making CDP useless) then reading show cdp neighbors.
Cloud0 wasn't actually empty — it was a collapsed cluster hiding a real router (Router3, a CGR1240) that needed its own IP config. Also that device has no routable Ethernet port — routing had to go through its Vlan1 SVI instead of a normal interface, which isn't how a standard 1841 router works.
A placeholder address (<cloud/ISP next-hop address>) got pasted literally into the ip route command, silently failing to parse — the rest of the NAT config went through fine, so the default route being missing wasn't obvious until show run | include ip route came back empty.
Switch-to-switch trunk port was configured as access mode on both switches instead of trunk, triggering a native VLAN mismatch (CDP flagged it directly). Both ends had to be converted to trunk with matching native VLAN.
VLANs existed on the trunk but not in each switch's local VLAN database — Switch0 never had VLAN 20 defined, Switch1 never had VLAN 10, so show interfaces trunk showed the VLAN "allowed" but not "active" until the missing vlan entries were added on each switch.
VLAN access commands were applied to the wrong physical ports — the original config assumed Fa0/1–2 were PC ports, but those were actually the router uplink and switch-to-switch trunk. Real PC ports (Fa0/3–4) were still sitting in default VLAN 1, so DHCP broadcasts never reached the switch's forwarding path toward the router at all. Found by hovering over the actual cables in Packet Tracer's topology view.
DHCP still failed even after every path issue was fixed, because Packet Tracer's DHCP radio button only sends a new DHCPDISCOVER on the transition into DHCP — it doesn't retry automatically. Had to toggle to Static and back to force a fresh request once the underlying network was actually correct.
